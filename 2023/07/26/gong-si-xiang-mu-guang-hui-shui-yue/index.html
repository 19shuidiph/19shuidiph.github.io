光辉税月

核心业务需求

在已有月报等类型的基础上，完成法律法规和案例风险的发布，查询。其中法律法规库的新增字段比较多

上传一个法律法规的压缩包，里面有excel文件和一个包含很多html文件的文件夹，完成解析，excel里每一条要对应到html，并且插入MySQL,MongoDB和搜索引擎。

其中MySQL记录一些规范的字段，MongoDB存储的是文件的内容content(很长，可能有一千字)，solr作为存储引擎，存储上面一些已经存在的内容，比如content,title,或者一些其他字段，主要是便于分词查找。

model

KbKlLawExt.java

针对于kb_kl_law_ext这个扩展表的对应类，扩展的是原来的kb_kl_ver表。

包括一个自增主键，一个kb_kl_ver的主键，还有九个法律法规表的特有字段。其中有一个叫发行日期(Date postDate);

KbDataDic.java

针对于kb_data_dic这个表对应的类，用MyBatisX生成

表的内容

Regulation.java

这个类是为了对接Excel表的数据，每一个属性对应于Excel表中的一列，方便excel快速导入数据库

除了excel的各列数据外，Regulation.java还新增了两个字段，分别是kb_kl的主键和kb_kl_ver的主键。（因为后续也要插入这两张表（其实是三张表，还有个kb_kl_law_ext,不过其核心主键也是kb_kl_ver)

RegulationRowExcel.java

为Regulation的子类，继承自他

扩展了一个属性字段-excelRow, 数据在excel中对应的行数

在检查标题的时候被赋予值。

mapper

所有的mapper.xml都遵循规范在resources的对应的同名目录下

KbKlLawExtMapper.java/xml

主要是根据KbKlLawExt直接MyBatisX生成的，其中也添加了一个按照kb_kl_ver来进行查询（selectOne)，还有一个回显的查询（也是根据verId)。

因为是回显页面，回显的应该是汉字，而非数据库存的对应的数字，所以这个回显查询的sql里查询了别的表。

dic_name是汉字(比如北京，上海)，dic_value是对应的数字（比如1,2)

<select id="selectDictValueByVerId" parameterType="integer" resultMap="BaseResultMap">
        select t.KL_LAW_ID,
               t.KL_VER_ID,
               (select dic_name from kb_data_dic d where d.dic_type = 'rule_file_type' and d.dic_value = t.FILETYPE) FILETYPE,
               (select dic_name from kb_data_dic d where d.dic_type = 'rule_level' and d.dic_value = t.REGULATORY_LEVEL) REGULATORY_LEVEL,
               (select dic_name from kb_data_dic d where d.dic_type = 'province' and d.dic_value = t.PROVINCE) PROVINCE,
               (select dic_name from kb_data_dic d where d.dic_type = 'city' and d.dic_value = t.CITY) CITY,
                POST_DATE,
                FILE_NUMBER,
                ISSUING_DEPARTMENT,
               (select dic_name from kb_data_dic d where d.dic_type = 'rule_file_valid' and d.dic_value = t.FILE_VALIDITY) FILE_VALIDITY,
               t.FAILURE_SCENARIO
        from kb_kl_law_ext t
        where t.KL_VER_ID = #{verid}
    </select>

KbDataDicMapper.java/xml

除了MyBatisX自动生成的外，还有两个核心业务

一个是得到列表，比如得到全国所有省份的列表

List<String> getTypesList(String s1);

KnowledgeMapper.java/xml

写了一个批量插入，后来因为MyBatis版本不支持，3.3版本以上才支持批量插入的主键回显。废弃了，KnowledgeVerMapper也是有个批量插入被废弃了

还写了一个根据标题查询。

List<String> queryByTitle(List<String> list);

详细实现是：

 <select id="queryByTitle" parameterType="list" resultType="string">
        select kb_kl.KL_TITLE from kb_kl where kb_kl.KL_TITLE in
        <foreach collection="list" item="item" separator="," open="(" close=")">
            #{item}
        </foreach>
 </select>

这个是为了支持excel插入的业务，用来返回excel中有没有与数据库中重复的

比如excel中有 1.1财政部发文XXX 1.2 国务院发文XX

数据库中有1.1财政部发文XXX, 则会将其返回，这样就会给前端报重复的错误，不进行再插入了。

service

RegulationService/impl

没有对应的RegulationMapper类，也没有单独的数据库表。主要是将excel接受到的数据写入到kb_kl,kb_kl_ver,kb_kl_law_ext三张表中，还有搜索引擎中。

每一条regulation对应于excel中的一行

Regulation中有两个函数，一个是insert,负责将这一小条regulation法规插入数据库和solr中

HashMap<String,Integer> insert(Regulation regulation, String dirPath, long countSolr, HashMap<String, String> hashmapOfFiveType) throws Exception;

另一个是checkTitle，负责检查我excel里所有的数据，是否与数据库中有重复的，返回值应该包括哪一行重复

List<RegulationRowExcel> checkTitle(List<Regulation> resultDataList);

insert函数

首先来说insert函数，这个函数的核心目的是调用

knowledgeService.doFirstSaveAndPublish(form,formExt1,tagList,attachmentList);

所以主要进行这个函数的四个参数的填充，主要是前两个参数，后两个参数我直接给的Null对象

第一个参数是常规的KnowledgeForm form，第二个参数是法律法规扩展的多余的8-10条信息，KbKlLawExtForm formExt1

因为我们是从隔壁的文件夹中获取内容数据，所以insert的第二个参数为dirPath路径，然后通过文件流的方式（inputStreamReader和BufferedReader)，读取到内容数据。

然后我们的insert函数的返回值是一个hashMap，在每个return hashmap之前，都要确保hashmap里有值。一般我给了三个值.

resultMap.put("失败",1);
resultMap.put("成功",0);
resultMap.put("耗时", 0);

insert的第三个参数countsolr用来计算时间，测试用的。

至于insert的最后一个参数hashmapOfFiveType，是由5个hashmap合并来的。

因为这五列数据没有重复的汉字，所以参数里面的五个hashMap可以合成为一个。反正hashmap都是以

key:汉字 -> value:数字值

checkTitle函数

首先通过List count1 = knowledgeMapper.queryByTitle(titleList);得到所有重复的文章的标题。

然后根据这些标题去找excel里是哪几行标题，得到具体的一些行数。

KnowledgeService/impl

对于KnowledgeService来说，只是在doFirstSaveAndPublish和doNotFirstSaveAndPublish里加了一个参数(KbKlLawExtForm formExt1)。当然其他调用此函数的地方，要多加一个null的参数。

对于函数内部而言，先判断formExt1 是否为null, 不为null则说明要插kb_kl_law_ext表。

这里要注意，如果文件号为空，默认值为null,应该把他赋值为空字符串，不然前端展示出来不好看。

KbKlLawExtService/impl

这里只有一个函数，根据verid获得kb_kl_law_ext的数据，即根据他的主表kb_kl_ver的主键获得数据。

这里注意一下判断verid是否为空就行。

ISearchService/impl

这边新增的业务是返回法律法规库和案例风险库的solr查询

两个函数分别满足法律法规库和案例风险库

函数 的参数是一些搜索条件，然后把这些条件整理一下喂给initDocQueryParam函数得到param,再根据param调用solr的globalSearchService得到结果列表。

因为initDocQueryParam接受的参数变化了，所以写了一个同名的重载函数。里面内容就是把参数喂给docQueryLawExtParam，再返回他。

IKIViewService/impl

这边只做了一个回显的操作，把法律法规查出来，显示给前端。因为是回显，要做一些内容的改变，比如注意日期，null等。

这里用到了mongodb来获取内容content。因为我们的内容字段是存在了mongodb中。

DocQueryLawExtEntity

这个类继承自DocQueryEntity,是和查询相关的。因为查询条件里有根据日期开始时间和结束时间进行一个范围查询，所以这边多了两个Date。当然还有那八个扩展属性(因为也有可能被查询）

@Field("postDate")
private Date postDateStart;

@Field("postDate")
private Date PostDateEnd;

@Field("postDate")
private Date postDate;

因为前面都加了@Field，所以这边也加了，这好像会导致返回值中上面三个属性返回值都是一样的，不过并不影响业务正确性。

DocQueryLawExtParam

这个类继承自DocQueryParam,由Param可以知道这里面就是一些查询的参数。只不过因为DocQueryParam这个原来的类里（因为需求是全量查询）只有一个简单的关键字String keyword，根本不能满足需求。所以要在这个DocQueryLawExtParam中加很多查询字段，包括但不限于八个新增字段，标题，内容，业务类型。而且查询参数里是开始时间和结束时间，这里也要有。

private List<DocQueryEntity> docList;

utils

这边只有一个类，判断对象是否为空（通过判断对象的所有属性）

public class CheckObjNull {
    public static boolean checkObjAllFieldsIsNull(Object object) {
        if (null == object) {
            return true;
        }
        try {
            for (Field f : object.getClass().getDeclaredFields()) {
                f.setAccessible(true);
                if (f.get(object) != null && StringUtils.isNotBlank(f.get(object).toString())) {
                    return false;
                }
            }
        } catch (Exception e) {
            e.printStackTrace();
        }
        return true;
    }
}

使用场景：

如果你新new了一个对象，则这个对象是不为null的，即使他的属性都为null,你要通过上述函数来判断。

web-(大部分改的都是controller)

DocQueryLawExtVO

先说vo，这边已经说过了。新加了一个DocQueryLawExtVO extends DocQueryLawExtEntity.

vo就是查询回显的。新增了一个属性dataStr, 给前端返回2023-07-27这样的字符串

KbKlLawExtForm

再说form层，扩展自KbKlLawExt类（这个类对应于kb_kl_law_ext表）

新增了一个postDateStr用来把date转换为字符串进行传递

新增了一个provinceLaw，因为kb_kl表有一个省份province.这属于当时建kb_kl_law_ext的时候没有考虑周全。

fore/SearchController

这里只写了doSearch4Law函数。只是调用了searchService的对应方法。

KnowledgeDocCtrl

修改了两个地方，新增了一个地方。

修改了编辑回显页面。因为法律法规库会多一些字段，要给他回显回去。

修改了单篇发布功能，如果类型不为法律法规库，要给扩展字段置为Null
