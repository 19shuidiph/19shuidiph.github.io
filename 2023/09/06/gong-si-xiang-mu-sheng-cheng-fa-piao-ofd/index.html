公司项目-生成发票OFD

test

FapiaoOfdClientTest这个类负责光伏，卷烟和普通电票的模板。

ConstructionFapiaoOfdClientTest这个类负责建筑服务的模板

RealEstateBusinessLeasingFapiaoOfdClientTest这个类负责不动产的模板

CargoTransportationFapiaoOfdClientTest这个类负责货物运输的模板

其中建筑和不动产，根据业务规则，只能生成一行货物明细。

这四个类只要起到填充数据的作用。当然在真实的业务场景里，是直接调用下面这行代码

FapiaoOfdClient.getInstance().createOfd(eInvoiceDomain,zos);

最核心的就是createOfd函数了。第一个参数是有数据的实体类，第二个参数是输入输出流。

流程

1.test进行测试，调用createOfd方法

2.进入FapiaoOfdClient类，根据传入的类型不同，选择调用不同的createOfd方法。可以点进beforeCreateOfd()的具体的createOfd()进行查看。

3.进入ofdClient包下的一个类，根据发票的商品数量，选择适合自己的函数

4.如果小于8条商品，会进入createOfdlt8（）函数。其中最核心的是这俩方法

// Doc_0/Tpls/Tpl_0/Content.xml  票面模板
new TplContentXmlCreater9(invoiceInfoDomain, configuration, zos).create();

// Doc_0/Pages/Page_0/Content.xml  票面内容
new PageContentXmlCreater9(invoiceInfoDomain, configuration, zos).create();

可以通过在其中拼接xml修改发票的红色模板和黑色内容

generator

下面可以得到发票生成的结果

core

里面有一些常量和对齐方式

creater

在creater中我们只关注doc里的page和tpl。

因为我们都是生成的单页发票，再进行拼接。所以annot,tag,以及一些其他的类，就不用碰了。

Tpl1对应Page1, 2 对应2 ，以此类推。 目前这个版本写了14个。

前6个是对应于普通电票的模板。7是不动产，8是建筑服务，9-14是交通运输服务。9-14就是1-6复制过来修改了一下。这些在ofdClient包下的类中可以找到对应关系.

模板1-小于8行的商品，单页，无小计，无分页，有备注

模板2-大于8行但是小于41行的商品，单页，无小计，无分页，有备注

模板3-47行的商品，单页，有小计，有分页，无备注。 后来升级为 大于8行且不是最后一页

模板4-被模板3取代了，已废弃，但是懒的删了。

模板5-小于8行，最后一页，有小计，有分页，有备注。

模板6-大于8行，最后一页，有小计，有分页，有备注。由模板2改造得到。

模板9-14以此类推，不过根据具体业务，数字不再是8和41，可能是4或37或3之类的。

domain

一些买方，卖方和其他的按钮，基础配置类之前就有了，不做详细说明。

本版本主要调整了核心实体类和商品实体类。

抽象出来了AbstractEInvoiceDomain类，让EInvoiceDomain,ConstructionEInvoiceDomain，RealEstateBusinessLeasingEInvoiceDomain,CargoTransportationEInvoiceDomain四个类都继承自抽象类。

如果商品明细有变动，也定义了对应的goods类，继承自抽象类AbstractGoodsInfoDomain

对于交通运输，是分上下两个列表的，下面的运输列表，定义在CargoTransportation类中。

这样CargoTransportationEInvoiceDomain里面除了继承自抽象类，就只有两个List了，非常简洁。

枚举类Category没啥用，写着玩的，可以参考一下，可以得知分类信息。

ofdClient

下面的四个类长的差不多，以BasicServiceFaPiaoOfdClient为例

pageLong和pageShort两个参数对应于无备注的最大商品行数量和有备注的最大商品行数量

一般前者为47，后者可以调整。交通运输服务的pageShort就调小了一些。

createOfd的第一个参数用的抽象类这个父类去接，目的是兼容所有类。在这里要强转一下，不然用不了子类的一些特定的函数。

然后用不同的模板，首先看一页能不能搞定。不能搞定就调用getAmountArray函数进行分页，然后创建一个temp文件夹，根据分页结果调用不同的模板，生成对应的单页ofd,然后zos.close(),然后进行合并ofd,然后把这些单页的临时ofd删除掉（调用deleteDirectoryStream方法）

utils

之前版本写的一些工具包

FapiaoOfdClient.java

自定义了一个函数式接口CustomBiConsumer，主要是用来支持hashmap的key为string, value为函数指针。

在beforeCreateOfd中对hashmap进行了初始化，可以保证用到哪个类就调用哪个类的专属createOfd函数

通过反射机制invoiceInfoDomain.getClass().getSimpleName()得到类的真实类名（即子类的类名），然后根据类名这个map里的key，去执行map里的value对应的函数。

新增模板的建议

如果你想要新增模板，建议你的商品和发票类这俩实体类都继承自domain里的抽象类。

在beforeOfd函数里添加对应的map.put

再在ofdClient中添加对应的新的客户端类，再在creater下建立自己的tpl和page类。 都是抄前面的改一改就行。

最后可以在test中新建个类去做测试。
